# Translation Layer — 多语言指令集

## 核心思想

**指令集的关键词不应该由英语垄断。**

ws-lang 的翻译层（Translation Layer）是一个**确定性查找表**，
将任意自然语言的关键词映射到内部规范表示（英文）。

这不是翻译引擎——不需要 AI，没有歧义，零误差。

## 工作原理

### 用户编写时

用户可以完全用自己的母语编写指令集：

```ws
# 中文用户
任务 "日报" {
  步骤 拉取数据 { 能力: ws-db-query }
}

# 日文用户
タスク "日報" {
  ステップ 取得 { 能力: ws-db-query }
}

# 英文用户
task "daily" {
  step fetch { cap: ws-db-query }
}
```

### 前置程序处理流程

```
用户输入
  │
  ▼
检测语言 / 用户指定的语言参数
  │
  ▼
加载对应语言的翻译表（translations/zh.json）
  │
  ▼
将自然语言关键词→规范关键词
  （任务→task, 步骤→step, 能力→cap）
  │
  ▼
按规范关键词解析 KV Line Format
  │
  ▼
编译为 YAML 内部格式
```

### 翻译表格式

```json
{
  "任务": "task",
  "步骤": "step",
  "能力": "cap",
  "出错时": "on_error"
}
```

**约束**：
1. 翻译表是**单射**——每个关键词最多映射到一个规范词
2. 翻译是**确定性**的——给定输入永远得到相同输出
3. 翻译表是**可扩展**的——任何人都可以添加新语言

## 语言检测策略

### 方案一：显式声明（推荐）

用户或 LLM 在文件头或参数中声明语言：

```ws
@lang: zh

task: 日报
  step: 查询
    cap: ws-db-query
```

### 方案二：自动检测

前置程序分析关键字特征判断语言：

- 如果出现"任务"、"步骤" → 中文
- 如果出现"タスク"、"ステップ" → 日文
- 否则默认为英文

## 扩展新语言

只需创建一个翻译 JSON 文件，通过 PR 提交：

```json
{
  "task": "task",
  "step": "step",
  "cap": "cap",
  ...
}
```

**就能让数百万母语者用自己的语言编写 ws-lang 指令集。**
